## Round 1

### Lead
**Action:** SUBMIT_FOR_REVIEW

Phase 6 plan — scoped to deliver value without full N-agent refactoring.

**Key decisions:**

1. **Deferred N-agent support** — The codebase has `VALID_TURNS = {"lead", "reviewer"}` hardcoded in state.py, binary turn logic in watcher.py, and role-specific branching throughout skills. Full N-agent support requires significant refactoring. This phase focuses on features that work with the existing 2-agent model.

2. **Migration command** — `python -m ai_handoff migrate` for projects set up before Phase 1 (no ai-handoff.yaml). Detects existing setup, auto-generates config with defaults (or detected names from docs), offers to re-run setup for template substitution.

3. **Model pattern matching** — Implements the design doc's `model_patterns` field for agent self-identification. Case-insensitive substring matching, `AI_MODEL_ID` env var as identifier source, validation for pattern overlaps.

4. **Config validation** — New `ai_handoff/config.py` module with `validate_config()` that enforces structure (lead + reviewer required, names non-empty, patterns are string lists).

5. **Agent identity env var** — `AI_HANDOFF_AGENT` can explicitly set which agent the current process is, useful for automation and tmux sessions.

**Out of scope:** 3+ agents, multiple leads/reviewers, agent specialties, `id` field, skills directory relocation, arbiter config, Windows sync-skills.

### Reviewer
**Action:** REQUEST_CHANGES

Good scope choices. I have two blocking gaps to resolve before implementation:

1. **Agent identity flow is underspecified and currently unused.** The plan adds `identify_agent()` and `AI_HANDOFF_AGENT`, but doesn’t define where that identity is actually used. The watcher doesn’t need identity (it just reacts to `handoff-state.json`), so setting `AI_HANDOFF_AGENT` there won’t change behavior. Please update the plan to:
   - Define the identity resolution order (e.g., `AI_HANDOFF_AGENT` → `AI_MODEL_ID` patterns → unknown) in `identify_agent()`, and
   - Specify the concrete integration point(s) that will call `identify_agent()` (e.g., a helper used by skills/CLI entry points that need to know their role, or a new `ai_handoff.identity` utility used by watchers/tools), and what happens on ambiguity.

2. **Config validation vs pattern overlap is inconsistent.** Plan says “warn on pattern overlap,” but the design doc treats overlaps as a validation error (to avoid ambiguous identity). Please decide and document the intended behavior (error vs warning) and how it affects `identify_agent()` (e.g., refuse to identify, return None, or raise).

Non-blocking suggestions:
- Consolidate config parsing (there are currently separate parsers in `cli.py`, `setup.py`, and `watcher.py`); consider routing through `config.py` to avoid divergence.
- Migration backups: specify where backups go and which files are backed up before running `setup` (e.g., timestamped `ai-handoff-backups/`).

## Round 2

### Lead
**Action:** SUBMIT_FOR_REVIEW

Revised the plan significantly based on feedback:

**Blocking item 1 resolved: Removed agent identity feature entirely.**
You're right — `identify_agent()` and `AI_HANDOFF_AGENT`/`AI_MODEL_ID` have no clear integration point in the current architecture. The watcher uses pane position, skills are markdown instructions. These features only become useful with N-agent support (future phase). Moved to Out of Scope with explicit rationale.

**Blocking item 2 resolved: Pattern overlap is an error, not a warning.**
Updated `validate_config()` to return an error on pattern overlap (e.g., "code" and "codex" could match the same identifier). This prevents ambiguous configurations upfront.

**Non-blocking suggestions incorporated:**

1. **Centralized config parsing** — New `ai_handoff/config.py` module with:
   - `read_config(path)` — single implementation with PyYAML fallback
   - `validate_config(config)` — structure validation + pattern overlap detection
   - `get_agent_names(config)` — helper for extracting names
   - All modules (`cli.py`, `setup.py`, `watcher.py`, `server.py`) updated to use it

2. **Migration backups** — Specified location: `ai-handoff-backups/YYYY-MM-DD_HHMMSS/templates/`. Only `templates/` directory is backed up (the files that get re-generated by setup).

**Scope is now:**
1. `migrate` command with auto-detection, dry-run, and backups
2. Centralized config module with validation
3. `model_patterns` field accepted (forward-compatible, not actively used)

Plan updated at `docs/phases/migration-and-advanced-features.md`.

### Reviewer
**Action:** REQUEST_CHANGES

One blocking doc consistency issue:

1. **Summary/What/Why still claim “model pattern matching for agent self-identification.”** The plan now explicitly moves self-identification out of scope, and only keeps `model_patterns` as a forward-compatible schema/validation field. Please update the Summary (and any other top-level phrasing) to reflect the revised scope so the plan doesn’t promise a removed feature.

Non-blocking suggestion:
- In the migrate auto-detect section, consider names with spaces/punctuation (regex currently shows `\w+` in the example). At least note the intended parsing behavior or fallback if names aren't single tokens.
